Zenoh.jl

Zenoh is a pub/sub/query protocol that unifies data in motion, data at rest, and computation over one location-transparent key/value space. Applications talk to keys arranged in a /-separated hierarchy; the network routes a put toward every matching Subscriber, serves a get from every matching Queryable or storage, and discovers peers and routers automatically. See the Zenoh abstractions manual for the full vocabulary — keys, key expressions, selectors, values, publishers, and the rest.

Zenoh.jl is a typed Julia wrapper over zenoh-c, the official C binding for the Rust Zenoh implementation. Each Zenoh entity — a Session, a Publisher, a Keyexpr, a Queryable — is a Julia struct with Julia-native ergonomics: GC-managed lifetimes, errors raised as exceptions, and eager operations that return a value or throw. The current surface spans publication and subscription, queries and queryables, liveliness, queriers, matching, advanced pub/sub, shared memory, and scouting.

Installation

Zenoh.jl is hosted on GitHub and bundles its native dependency through a JLL, so no separate Zenoh install is required:

using Pkg
Pkg.add(url = "https://github.com/BenChung/Zenoh.jl.git")
using Zenoh

The first run compiles the package and loads libzenohc. To talk to other Zenoh applications you need either peer connectivity or a running zenohd router; the examples throughout this manual assume one is reachable.

A first session

using Zenoh

# Build a config: defaults, or from a JSON5 string / file / the ZENOH_CONFIG env var.
cfg = Config(; str = """{mode: "peer"}""")
cfg["connect/endpoints"] = ["tcp/localhost:7447"]   # JSON5-serialized insert

s = open(cfg)                 # eager: returns a Session or throws ZenohError
try
    @show isopen(s)
    @show zid(s)              # this session's 16-byte Zenoh ID
    @show router_zids(s)      # IDs of connected routers

    k = kexpr"demo/example"   # a Keyexpr via the string macro
    put(s, k, "hello")        # eager publish; throws on failure
finally
    close(s)                  # deterministic teardown; idempotent
end

The Getting Started page walks through this end to end; Sessions & Configuration covers the full configuration surface.

The wrapper model

Four conventions hold across every page of this manual, so each feature page can assume them.

GC-managed lifetimes

Every Zenoh entity is garbage-collected. An unreferenced Session, Publisher, or Config frees its resources on its own, with no explicit teardown required.

Deterministic close alongside GC

For resources whose teardown order matters, close (or undeclare for declared entities) tears the resource down immediately on the calling task. A Session shuts down gracefully and drains its background runtime on that task, where teardown order is predictable, rather than leaving it to the garbage collector. Closing the same entity again is a no-op, and GC remains a safety net for entities never closed explicitly.

A closed session throws on use

Any operation on a closed Session raises an ArgumentError. close is idempotent, but operations are not retryable after it.

Errors as exceptions

Every operation either returns its result or throws a ZenohError. A successful call returns the wrapped object or nothing; there is no status code to inspect.

Eager operations, no builder layer

Every Zenoh.jl operation runs immediately. open, put, Publisher(s, k), get, and their siblings perform the operation on the spot and return the result or throw — there is no separate builder or resolve step.

Some features (advanced pub/sub, the structured serializer) come from the zenoh-ext extension rather than the core protocol; each such page flags this where it applies.

Implementation notes

The conventions above are the user-facing contract; this section records how the wrapper realizes them over zenoh-c, and where it diverges from the Rust API. Each manual page carries its own "Mapping to Zenoh, Rust, and C" section for its domain.

Ownership categories

zenoh-c splits its types into four categories: owned (z_owned_*_t, holding a resource that must be dropped), loaned (z_loaned_*_t, a borrow), moved (z_moved_*_t, consumed by the callee), and view (z_view_*_t, a pointer into external data). Every owned C handle is wrapped in a Julia struct holding a Base.RefValue, with a finalizer that drops it on garbage collection — RAII reaching Julia. src/core/ownership.jl generates the lifecycle verbs at module load: it scans LibZenohC for every z_owned_*_t, finds the matching moved and loaned types, and dlsyms the corresponding C functions to emit the applicable _loan/_loan_mut/_move/_take/_drop verbs — only those whose C symbols exist. View types own nothing, so they need no verbs and are never dropped.

Close and error mechanics

A Session carries a closed flag and cached SHM state in mutable Ref cells beyond the bare z_owned_session_t, and every declare/put/get checks that flag before calling into C. close calls z_close then drops the handle (z_session_drop) — both on the caller's task, so Zenoh's runtime drains where ordering is predictable rather than on the finalizer thread. Fallible C calls return a z_result_t code; the wrapper throws ZenohError on any non-Z_OK result. The Rust API defers operations through a Resolvable builder finished with .wait()/.await; the pre-resolved C API (z_open, z_put, z_declare_publisher) and Zenoh.jl run on the spot.

Prefix taxonomy

The wrapped C symbols carry one of three prefixes:

  • z_ — the core API, shared with zenoh-pico (z_open, z_put, z_declare_subscriber).
  • zc_ — zenoh-c specific (zc_config_from_env, zc_config_to_string).
  • ze_ — zenoh-ext, outside the core protocol.
Not core Zenoh

Some features sit outside the core protocol. Advanced Pub/Sub wraps zenoh-ext (ze_* symbols), as does the structured serializer on Payloads & Serialization.

ZRef has no Zenoh counterpart

ZRef — a typed, transport-agnostic payload handle — is a Zenoh.jl construct with no direct Zenoh, Rust, or C equivalent.
